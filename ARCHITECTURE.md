# Архитектура Report Platform

## 1. Компоненты и потоки данных

### Высокоуровневая схема

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Frontend  │────▶│   Backend   │────▶│   Worker    │
│  (React)    │◀────│  (Express)  │◀────│  (Bull)     │
└─────────────┘     └─────────────┘     └─────────────┘
                         │                     │
                         ▼                     ▼
                  ┌─────────────┐     ┌─────────────┐
                  │ PostgreSQL  │     │    Redis    │
                  │   (БД)      │     │   (Очередь) │
                  └─────────────┘     └─────────────┘
```

### Компоненты

1. **Frontend (React + TypeScript)**
   - Веб-интерфейс для взаимодействия с платформой
   - Отображает список отчётов, историю запусков, позволяет запускать генерацию
   - Общается с Backend через REST API

2. **Backend API (Node.js + Express)**
   - REST API для управления отчётами и запусками
   - Принимает запросы на генерацию, создаёт задачи в очереди
   - Предоставляет статусы и результаты
   - Интегрирует Bull Board для мониторинга очередей

3. **Worker (Node.js + Bull)**
   - Обрабатывает задачи из очереди Redis
   - Выполняет генерацию отчётов (XLSX, PDF)
   - Обновляет статусы в БД
   - Получает данные через абстракцию источников `DataSourceProvider`

4. **PostgreSQL**
   - Хранит метаданные шаблонов отчётов (ReportTemplate) и сгенерированных отчётов (Report)
   - Схема: ReportTemplate (1) → Report (N)

5. **Redis**
   - Очередь задач для асинхронной обработки
   - Хранение состояния задач Bull

6. **Docker Compose**
   - Оркестрация всех сервисов для локального запуска

### Поток данных при генерации отчёта

1. Пользователь нажимает "Generate Report" во фронтенде.
2. Frontend отправляет `POST /api/report-templates/:id/generate` в Backend.
3. Backend создаёт запись `Report` со статусом `PENDING` в PostgreSQL.
4. Backend добавляет задачу в очередь Bull (Redis).
5. Worker забирает задачу из очереди, меняет статус на `PROCESSING`.
6. Worker выполняет генерацию (заглушка или реальная логика).
7. По завершении Worker обновляет статус на `COMPLETED` (или `FAILED` при ошибке).
8. Frontend периодически опрашивает статус через `GET /api/reports` и отображает результат.
9. Пользователь может скачать готовый отчёт через `GET /api/reports/:id/download`.

### Источники данных (разнородность)

В воркере используется интерфейс `DataSourceProvider` (`backend/src/dataSources/reportDataSources.ts`), который отделяет генерацию отчёта от конкретного способа получения данных.

- `PostgresSalesDataSource` — читает данные из PostgreSQL через Prisma (в прототипе для шаблонов типа `Sales`)
- `ExternalAnalyticsDataSource` — имитирует внешний API (mock external API, в прототипе для шаблонов типа `Analytics`)

За счёт этого новый источник можно подключить без изменений API-контрактов и UI: достаточно добавить новый provider и выбрать его в `reportJob`.

## 2. Как добавить новый отчёт – пошаговая инструкция для разработчика

### Шаг 1: Определить данные и источник

Решите, какие данные будет содержать отчёт и откуда они будут браться (БД, внешний API, файл). Для прототипа можно использовать моковые данные.

### Шаг 2: Создать генератор (опционально)

Если отчёт требует нового формата (например, CSV), добавьте модуль в `backend/src/reportGenerators/`. Пример:

```typescript
// backend/src/reportGenerators/csvGenerator.ts
export async function generateCsv(data: any[]): Promise<Buffer> {
  // реализация
}
```

### Шаг 3: Добавить источник данных через DataSourceProvider

Создайте новый provider в `backend/src/dataSources/reportDataSources.ts`, реализующий интерфейс `DataSourceProvider`.

```typescript
class CsvFileDataSource implements DataSourceProvider {
  async getData(): Promise<any[]> {
    // чтение/парсинг CSV
    return [];
  }
}
```

Затем подключите provider в `getDataSourceProvider(...)` (например, по `template.type`), чтобы `reportJob` начал использовать новый источник.

### Шаг 4: Добавить шаблон отчёта в базу данных

Создайте миграцию или добавьте запись через seed-скрипт (`backend/scripts/seed.ts`):

```typescript
await prisma.reportTemplate.create({
  data: {
    name: 'My New Report',
    description: 'Описание нового отчёта',
    format: 'PDF',
    type: 'Finance',
  },
});
```

### Шаг 5: Протестировать

Запустите платформу, убедитесь, что отчёт появился в интерфейсе и генерация работает.

### Шаг 6: Документировать

Обновите документацию, если необходимо.

**Важно**: Вся бизнес-логика изолирована в `reportJob.ts`, `reportGenerators/` и `dataSources/`. Добавление нового отчёта обычно не требует изменений API, очередей и фронтенда (если не нужны новые поля/фильтры в UI).

## 3. Принятые решения и альтернативы

В этом разделе подробно рассматриваются ключевые технологические решения, принятые при построении платформы, а также альтернативы, которые могли бы быть использованы. Каждое решение обосновано с точки зрения требований прототипа, скорости разработки, масштабируемости и соответствия современным best practices.

### 3.1. Бэкенд-фреймворк: Express

**Выбрано**: Express как минималистичный фреймворк для REST API.

**Альтернативы**:
- **NestJS**: Полнофункциональный фреймворк с DI, модулями, встроенной поддержкой WebSockets, GraphQL. Отличный выбор для крупных проектов, но добавляет сложность и boilerplate для прототипа.
- **Fastify**: Более производительный, чем Express, с автоматической валидацией через JSON Schema. Однако меньше middleware и сообщество.
- **Koa**: Современный наследник Express с async/await из коробки, но менее распространён.
- **Hapi**: Конфигурационный подход, популярен в enterprise, но более verbose.

**Почему Express**:
- **Минимализм и гибкость** – позволяет быстро создать API без навязывания структуры.
- **Огромное количество middleware** (cors, helmet, morgan, passport) для добавления функциональности по мере необходимости.
- **Простота интеграции** с Bull Board, Prisma, Swagger.
- **Широкая известность** – легко найти разработчиков и примеры кода.
- **Достаточно для прототипа** – все требования (маршруты, middleware, обработка ошибок) реализуются с минимальными усилиями.

### 3.2. Очередь задач: Bull + Redis

**Выбрано**: Bull (библиотека для Node.js) с Redis в качестве брокера.

**Альтернативы**:
- **RabbitMQ** (AMQP): Классический брокер сообщений с продвинутыми возможностями (обменники, routing keys, dead letter queues). Подходит для сложных workflow, но требует отдельного сервиса и более сложной настройки.
- **Apache Kafka**: Распределённый log, идеален для потоковой обработки и событийных архитектур. Избыточен для простых фоновых задач, операционно тяжёл.
- **AWS SQS / Google PubSub**: Fully managed очереди в облаке. Устраняют необходимость поддерживать инфраструктуру, но создают vendor lock‑in и сложности при локальной разработке.
- **Agenda** (на основе MongoDB): Использует MongoDB как хранилище, не требует отдельного Redis. Удобна, если уже используется MongoDB, но менее производительна и имеет меньше функций.
- **Bee‑Queue**: Облегчённая альтернатива Bull, но с ограниченным функционалом и меньшей активностью развития.
- **In‑memory очередь** (например, `queue` из `async`): Самая простая, но не сохраняет состояние при перезапуске и не масштабируется на несколько процессов.

**Почему Bull + Redis**:
- **Специально разработана для Node.js** – идеальная интеграция, Promise‑based API.
- **Богатый функционал** из коробки: повторные попытки с backoff, приоритеты, задержки, события прогресса, пауза/возобновление.
- **Мониторинг через Bull Board** – встроенный UI для просмотра очередей, что критически важно для отладки и наблюдения в прототипе.
- **Redis как хранилище** – быстро, надёжно, поддерживает persistence. Redis также может использоваться для кэширования и сессий в будущем.
- **Легко масштабируется** – можно запускать несколько воркеров на разных машинах, слушающих одну очередь.
- **Активное сообщество** – Bull широко используется в production, есть множество примеров и статей.

### 3.3. База данных: PostgreSQL + Prisma

**Выбрано**: PostgreSQL как реляционная СУБД с Prisma в качестве ORM.

**Альтернативы**:
- **MySQL**: Похожа на PostgreSQL, но historically менее развита в части advanced типов данных и JSON поддержки. Широко распространена, но для новых проектов часто выбирают PostgreSQL.
- **MongoDB**: Документ-ориентированная БД, предлагающая гибкую схему. Хороша для данных с переменной структурой, но отсутствие JOIN и транзакций (до недавних версий) усложняет реляционные связи (шаблоны → отчёты).
- **SQLite**: Встраиваемая БД, проста в использовании, но не подходит для многопользовательского доступа и параллельных запросов в продакшене.
- **CockroachDB**: Распределённая SQL БД, совместимая с PostgreSQL. Отличный выбор для глобального масштабирования, но избыточен для прототипа.
- **Supabase / Firebase**: Backend‑as‑a‑Service, предоставляющие готовые API. Ускоряют разработку, но ограничивают контроль и миграции.

**Почему PostgreSQL**:
- **Надёжность и ACID** – гарантирует целостность данных, что важно для финансовых отчётов и аудита.
- **Богатые возможности** – JSONB для гибких данных, полнотекстовый поиск, геоданные, расширения (PostGIS, TimescaleDB).
- **Производительность** – эффективные индексы, оптимизатор запросов, параллельное выполнение.
- **Экосистема и инструменты** – множество клиентов, миграционных инструментов, облачных managed‑решений (AWS RDS, Google Cloud SQL).

**Почему Prisma**:
- **Типобезопасность** – генерирует TypeScript‑типы на основе схемы, предотвращая ошибки в рантайме.
- **Интуитивный API** – `prisma.report.findMany({ where: { status: 'COMPLETED' } })` читается как обычный JavaScript.
- **Миграции** – `prisma migrate dev` создаёт и применяет миграции на основе изменений схемы.
- **Интеграция с экосистемой** – работает с Next.js, GraphQL, Serverless.
- **Производительность** – использует Prepared Statements, connection pooling, эффективные запросы.

### 3.4. Фронтенд: React + TypeScript + Material‑UI

**Выбрано**: React с TypeScript и Material‑UI (MUI) в качестве UI‑библиотеки.

**Альтернативы**:
- **Vue.js** + **Vuetify / Quasar**: Более gentle learning curve, отличная документация. Однако React имеет большее сообщество и лучше интегрируется с TypeScript.
- **Angular** + **Angular Material**: Полнофункциональный фреймворк с DI, RxJS, мощным CLI. Подходит для enterprise, но излишне сложен для прототипа.
- **Svelte / SvelteKit**: Новый подход, компилирующий компоненты в чистый JS, меньше boilerplate. Менее зрелая экосистема.
- **Next.js** (React‑фреймворк): Предоставляет SSR, SSG, API routes. Отличный выбор для production, но добавляет абстракции, которые не нужны в прототипе.
- **State‑менеджеры** (Redux, Zustand, MobX): Усложняют состояние без необходимости – в прототипе достаточно React Context и локального состояния.

**Почему React + TypeScript + MUI**:
- **React** – де‑факто стандарт для современных SPA, огромное сообщество, богатая экосистема (React Router, React Query, Formik).
- **TypeScript** – обеспечивает типизацию пропсов, состояния, API‑ответов, значительно снижая количество runtime‑ошибок.
- **Material‑UI** – предоставляет готовые, доступные и стилизованные компоненты (таблицы, карточки, модалки, кнопки), что ускоряет разработку UI.
- **Feature‑Sliced Design** – выбранная архитектурная методология для фронтенда позволяет чётко разделить ответственность (`app`, `pages`, `widgets`, `features`, `entities`, `shared`), что облегчает поддержку и масштабирование.

### 3.5. Контейнеризация и оркестрация: Docker Compose

**Выбрано**: Docker Compose для локального запуска всего стека (PostgreSQL, Redis, бэкенд, воркер, фронтенд).

**Альтернативы**:
- **Kubernetes (Minikube / Kind)**: Промышленный стандарт оркестрации, но избыточен для локальной разработки прототипа.
- **Podman**: Бес‑демонная альтернатива Docker, совместимая с Docker Compose. Менее распространена.
- **Ручной запуск** (установка всех сервисов на хосте) – трудоёмко, непереносимо между разработчиками.
- **Terraform + Cloud‑сервисы** – provisioning инфраструктуры в облаке, но сложно для локального тестирования.

**Почему Docker Compose**:
- **Простота** – один файл `docker‑compose.yml` описывает все сервисы, сети, тома.
- **Переносимость** – гарантирует, что каждый разработчик запускает идентичное окружение.
- **Быстрый старт** – `docker‑compose up --build` поднимает всю платформу за несколько минут.
- **Изоляция** – каждый сервис работает в своём контейнере, нет конфликтов версий.
- **Подготовка к продакшену** – те же образы можно использовать в CI/CD и production‑оркестраторах.

### 3.7. Архитектурный стиль: «Тонкий бэкенд + отдельный воркер»

**Выбрано**: Разделение на два независимых сервиса – API (принимает запросы) и Worker (обрабатывает фоновые задачи).

**Альтернативы**:
- **Монолит с фоновыми задачами** (например, `setInterval` или `node‑cron` внутри того же процесса API). Проще в развёртывании, но блокирует Event Loop при долгих задачах и усложняет масштабирование.
- **Serverless функции** (AWS Lambda, Google Cloud Functions) для генерации отчётов. Автоматическое масштабирование, но холодные старты, ограничения времени выполнения, сложность локальной разработки.
- **Микросервисы** (отдельные сервисы для шаблонов, очередей, генерации, нотификаций). Максимальная гибкость, но огромная операционная нагрузка и overhead для прототипа.

**Почему разделение**:
- **Чёткое разделение ответственности** – API занимается только HTTP‑запросами, Worker – только фоновой обработкой.
- **Независимое масштабирование** – при росте нагрузки можно увеличить количество реплик воркера, не трогая API.
- **Отказоустойчивость** – падение воркера не влияет на доступность API (и наоборот).
- **Соответствие принципам микросервисов** – даже в прототипе демонстрируется подход, который легко эволюционирует в полноценную распределённую систему.

### 3.8. Генерация отчётов: заглушки вместо реальных библиотек

**Выбрано**: Заглушки `xlsxGenerator.ts` и `pdfGenerator.ts`, возвращающие mock‑данные.

**Альтернативы**:
- **Реализация с `exceljs` и `pdfkit`** – полноценная генерация XLSX и PDF. Требует дополнительных зависимостей и времени на реализацию.
- **Puppeteer** – для генерации PDF из HTML‑шаблонов (например, с использованием React). Мощный, но тяжёлый (запуск headless Chrome).
- **JasperReports / BIRT** – enterprise‑решения для отчётов, требующие отдельного сервера и сложной интеграции.
- **Google Sheets API / Microsoft Graph API** – облачные API для создания таблиц и документов. Vendor lock‑in и зависимость от интернета.

**Почему заглушки**:
- **Фокус на архитектуре** – цель прототипа – показать, как организовать поток данных, а не детали генерации файлов.
- **Быстрая разработка** – не нужно тратить время на настройку библиотек и обработку edge cases.
- **Демонстрация расширяемости** – заглушки чётко показывают, куда должна быть вставлена реальная логика. Разработчик, читающий код, сразу понимает, как заменить mock на реальную генерацию.
- **Соответствие принципу «думайте широко, реализуйте узко»** – прототип должен быть минимально достаточным для демонстрации концепции.

### 3.9. Логирование: Pino

**Выбрано**: Pino как высокопроизводительный логгер для Node.js.

**Альтернативы**:
- **Winston** – более популярный, с множеством транспортов, но менее производительный.
- **Bunyan** – похож на Pino, но менее активное развитие.
- **console.log** – простейший вариант, но отсутствие структуры, уровней логирования, ротации.

**Почему Pino**:
- **Высокая производительность** – минимальный overhead, что важно для высоконагруженных приложений.
- **Structured JSON‑логи** – легко парсятся системами сбора логов (ELK, Loki).
- **Экосистема** – транспорты для Sentry, Datadog, файловой ротации.
- **Простота настройки** – `const logger = pino()` и готово.

### 3.10. Принципы SOLID и чистая архитектура

**Выбрано**: Следование принципам SOLID и элементам чистой архитектуры для создания гибкой, поддерживаемой системы.

**Альтернативы**:
- **Монолитная спагетти‑архитектура** – всё в одном месте, быстрая первоначальная разработка, но невозможность масштабирования и высокий технический долг.
- **Жёсткие зависимости** – прямое использование конкретных реализаций вместо абстракций, что приводит к хрупкости и сложности тестирования.

**Почему SOLID и чистая архитектура**:
- **Single Responsibility Principle (SRP)** – каждый модуль имеет одну причину для изменения. Например:
  - `DataSourceProvider` отвечает только за получение данных.
  - `generateXlsx` / `generatePdf` отвечают только за генерацию файлов.
  - `reportJob` координирует процесс, но не содержит логики получения данных или рендеринга.
- **Open/Closed Principle (OCP)** – система открыта для расширения, но закрыта для модификации. Добавление нового источника данных (например, CSV‑файла) требует только создания нового класса, реализующего `DataSourceProvider`, без изменения существующего кода в `reportJob` или `getDataSourceProvider`.
- **Liskov Substitution Principle (LSP)** – подтипы могут заменять базовые типы. `PostgresSalesDataSource` и `ExternalAnalyticsDataSource` взаимозаменяемы везде, где ожидается `DataSourceProvider`.
- **Interface Segregation Principle (ISP)** – интерфейсы узко специализированы. `DataSourceProvider` содержит единственный метод `getData()`, что соответствует потребностям клиента (`reportJob`). Нет «жирных» интерфейсов с избыточными методами.
- **Dependency Inversion Principle (DIP)** – зависимости направлены на абстракции, а не на конкретные реализации. `reportJob` зависит от интерфейса `DataSourceProvider`, а не от конкретных классов. Это позволяет легко подменять источники данных, например, для тестирования (mock) или добавления новых источников.

**Элементы чистой архитектуры**:
- **Разделение слоёв**: хотя прототип не реализует полноценную чистую архитектуру, прослеживается чёткое разделение:
  - **Инфраструктурный слой**: Prisma, Bull, Express – детали реализации внешних сервисов.
  - **Слой приложения**: `reportJob` – оркестрация бизнес‑процесса.
  - **Домен**: `Report`, `ReportTemplate` – бизнес‑сущности (представлены в Prisma‑схеме).
- **Инверсия зависимостей**: ядро (бизнес‑логика) не зависит от деталей. Например, генератор отчётов не знает, откуда пришли данные – он получает их через абстракцию `DataSourceProvider`.
- **Тестируемость**: благодаря абстракциям можно легко писать unit‑тесты, подменяя реальные источники данных и генераторы mock‑объектами.

**Практические примеры из кода**:
1. **Добавление нового источника данных** – создать класс, реализующий `DataSourceProvider`, и добавить его в `getDataSourceProvider`. Никаких изменений в `reportJob` не требуется.
2. **Добавление нового формата отчёта** – создать функцию‑генератор и добавить условие в `reportJob`. Это изменение локализовано и не затрагивает другие компоненты.

**Итог**: Применение SOLID и чистых архитектурных принципов делает платформу легко расширяемой, тестируемой и готовой к эволюции. Это соответствует цели прототипа – показать, как система может расти без переписывания.

## 4. Что не сделали и почему

### 4.1. Аутентификация и авторизация

**Почему**: Требование прототипа — демонстрация платформы отчётов, а не система безопасности. В продакшене необходимо добавить JWT/OAuth2.

### 4.2. Хранение файлов в объектном хранилище (S3)

**Почему**: Упрощение. Вместо S3 результат «хранится» как mock-файл в памяти. В реальном проекте нужно интегрировать MinIO / AWS S3.

### 4.3. Полноценное логирование и мониторинг

**Почему**: Используется только Pino для логов. Нет метрик, алертов, Grafana. Для продакшена необходимо добавать.
